7 layers of the OSI

 

Application Layer (Layer 7)

Closest layer to the end user. Network services to applications. HTTP, FTP, SMTP, DNS

Presentation Layer (Layer 6)

Translates data: application <=> network format. Covers encryption, compression, and data translation. SSL/TLS, encryption formats.

Session Layer (Layer 5)

Manages sessions or connections between devices. Establishes, maintains, and terminates connections. API sessions, remote procedure calls.

Transport Layer (Layer 4)
  • Translating a domain name into an IP address 
  • Ensures reliable data transmission between devices. Responsible for flow control, error correction, and segmentation. TCP, UDP, ports.  
    • TCP: Banking info. Sending important data. Poor streaming quality; TCP 3-way handshake required: SYN, SYN-ACK, and ACK. Security+, quality wont be good, not inherently encrypt data, unless paired with additional encryption protocols like SSL/TLS .
    • UDP: Stream data ideal for broadcasting, real-time (voice,videostreaming) vulnerable to hackers, to general public, VoiP, lacks security features.
Network Layer (Layer 3)
  • Handles routing of data between devices. Determines the best path for data transmission. IP addresses, routers, packets.
  • IPSec: Connection slower, every IP package sent is encrypted.  highly secure environment, for sensitive data or remote access via VPNs.
Data Link Layer (Layer 2)
  • LLC and MAC address
  • Responsible for node-to-node data transfer. Detects and corrects errors in the physical layer. MAC address, switches, frames
Physical Layer (Layer 1) 
  • Transmission of raw bits
  • Deals with the physical connection between devices. Transmits raw bit streams over a physical medium (e.g., cables, wireless). Eth cables, fiber optics.
  • DHCP protocol used

Pegasus 

Spyware designed to infect an asset, gains access, provides remote control. It relies on zero-click exploits (no user interaction) or one-click attacks.

 

FemtoCell

Small cellular stations, used to improve coverage in limited area, they connect to a mobile provider’s network via the internet.

Target device <-> FemToCell <-> Cellular tower

To force a connection to FemToCell jam legitimate cell signals or overpower current signal with stronger from the asset. 

Available exploits: eavesdropping (when downgrading 5G->3G), data collection, malware installation, manipulate the data packets, inject malware payloads without the user clicking any link but establishing a network-level exploit.

Once connected to the FemToCell, exploit cellular protocol or the OS. Install Pegasus or similar spyware. Happens without the user’s interaction, combination of network-level exploits and payload delivery. The FemToCell asset needs to be close to the target asset. Physically nearby

 

 

Stingray

International Mobile Subscriber Identity (IMSI) catcher. Tracking and intercepting cellular communications. Stingray mimics cell towers, tricking mobile phones into connecting to them. 

Impersonates a Cell Tower broadcasting signals that make it appear as the strongest or closest cell tower. Mobile phones in the vicinity automatically connect to it, as phones are programmed to connect to the tower with the strongest signal. Once connected, captures phone's IMSI. The device can track the phone's location without needing GPS.

Based on the model, may be able to intercept calls having the capability to listen, intercept text messages, and data packets. Some can even force the target phone to downgrade from 4G/5G to 2G which uses weak encryption, easier to eavesdrop. Do not target specific individuals but rather collect data from all phones within range, affecting thousands of users.

Dual Bootable OS

Micro-sd cards are not directly recognized in BIOS use a external reader. To preview local partition in Kali installation interface, switch from RAID TO AHCI -> ms.config -> safe mode

RAID: Systems w multiple hard drives for performance combining physical disks into one logical volume.
AHCI: Simple storage ver. used for standard hard drives and SSDs.

Most assets have RAID mode enable. However, Linux may not recognize RAID partitions, not showing up your partitions when installing. AHCI compatible with Linux.

RAID to AHCI without preparation, may Windows crashed, and require reinstall W. A workaround to prevent crash:

In windows, CURRENT RAID settings. Enable Safe Mode: cmd-> msconfig -> Boot tab -> check the Safe Boot box. Apply and restart your system.

Now in BIOS -> SATA Configuration -> Change from RAID to AHCI -> Save and exit BIOS. This will boot W in Safe Mode because Safe Mode enabled. Windows automatically load the AHCI drivers. Once Safe Mode, cmd -> msconfig -> Boot tab and uncheck Safe Boot. Apply changes and restart. Booting NORMALLY in W under AHCI mode, without hazard. Proceed with Kali installation: Restart system and boot into Kali installer. Partitions should be visible since your system is in AHCI mode, compatible with Linux.

This lab will guide students through becoming a Certificate Authority (CA), creating certificates for a website, deploying them on an HTTPS server, and conducting basic experiments with certificate integrity.
Task 1: Becoming a Certificate Authority (CA)
Objective: Set up your environment to act as a Certificate Authority (CA).
Steps:
Copy the configuration file:
Run the following command to copy openssl.cnf to your current directory:
cp /usr/lib/ssl/openssl.cnf .
Create the required directories:
Run the following commands to set up directories specified in openssl.cnf:
mkdir demoCA cd demoCA mkdir certs crl newcerts touch index.txt serial echo 01 > serial
Generate a self-signed certificate:
Navigate back to your root project directory and run the following command to create self-signed certificates for the CA:
openssl req -new -x509 -keyout ca.key -out ca.crt -config openssl.cnf
Verify the CA's certificate:
Open and review the ca.crt file:
openssl x509 -in ca.crt -text
Task 2: Creating a Certificate for Your Personal Website (e.g., sam12345.com)
Before proceeding, please install and configure Apache server on your Kali Linux. Please refer to the following website: HTTP server configuration using Kali Linux | by Harsh Dubey | MediumLinks to an external site.
Objective: Create a certificate for your personal website using the format first name + first five digits of your student ID.
Step 1: Generate public/private key pair:
Generate the key pair with the following command:
openssl genrsa -aes128 -out server.key 1024
Step 2: Generate a Certificate Signing Request (CSR):
Run the following command to create a CSR, setting the common name to sam12345.com (replace with your name and ID):
openssl req -new -key server.key -out server.csr -config openssl.cnf
Step 3: Sign the CSR:
Adjust the openssl.cnf file to change the matching rules to policy_anything.
Run the following command to sign the CSR:
openssl ca -in server.csr -out server.crt -cert ca.crt -keyfile ca.key -config openssl.cnf
Task 3: Deploying the Certificate in an HTTPS Web Server
Objective: Set up a web server to use the signed certificate for sam12345.com (replace with your name and ID).
Step 1: Configure DNS:
Edit the /etc/hosts file and add the following line:
127.0.0.1    sam12345.com
Step 2: Configure the web server:
Combine the private key and certificate into a .pem file:
cp server.key server.pem cat server.crt >> server.pem
Step 3: Add the CA certificate to the browser:
Import ca.crt into Firefox and mark it as trusted to identify websites.
Step 4: Access the website:
Open https://sam12345.com:4433 (replace with your website) in your browser and verify that there is no certificate error.
Task 4: Deploying Certificate in an Apache-Based HTTPS Website
Objective: Deploy the signed certificate on an Apache-based HTTPS website for sam12345.com (replace with your name and ID).
Modify the Apache configuration:
Navigate to /etc/apache2/sites-available and open 000-default.conf.
Add the following VirtualHost entry:
<VirtualHost *:443>     ServerName sam12345.com     DocumentRoot /var/www/sam12345     SSLEngine on     SSLCertificateFile /path/to/server.crt     SSLCertificateKeyFile /path/to/server.key </VirtualHost>
Enable SSL and configure Apache:
Enable SSL with:
sudo a2enmod ssl
Test the configuration with:
sudo apachectl configtest
Enable the SSL site with:
sudo a2ensite default-ssl
Restart the Apache server:
sudo service apache2 restart
Task 5: Launching a Man-in-the-Middle (MITM) Attack
Objective: Simulate a MITM attack by setting up a malicious server and DNS spoofing for a target site like youtube.com.
Modify DNS on the victim machine:
Edit /etc/hosts on the victim's machine to point youtube.com to the attacker's IP.
Set up the malicious website:
Create a VirtualHost entry for the target website (youtube.com) and restart Apache.
Add the CA certificate to the victim's browser:
Add the attacker's ca.crt to the victim's browser to trick it into trusting the attacker's certificates.
Browse the target website:
Access youtube.com on the victim's browser and observe the connection.
Task 6: Launching a MITM Attack with a Compromised CA
Objective: Use the private key of a compromised CA to sign a certificate for a target website.
Generate and sign a certificate for the target site:
Generate a key and CSR for the target website (e.g., youtube.com), then sign it using the compromised CA's private key.
Set up the VirtualHost for the malicious website:
Configure Apache to serve the malicious website for youtube.com.
Test the attack:
Browse the target website and observe how the victim's browser now accepts the malicious certificate.
